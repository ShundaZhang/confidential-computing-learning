# 安全多方计算（MPC）

安全多方计算（Secure Multi-Party Computation, MPC）是一类密码学协议，允许多个参与方在不暴露各自原始输入的情况下，共同计算一个函数并得到结果。与 TEE 依赖硬件隔离不同，MPC 的安全性来自协议和密码学假设。

## 核心目标

给定多个参与方 `P1...Pn`，每方持有私有输入 `x1...xn`。MPC 希望计算 `y = f(x1, x2, ..., xn)`，同时满足：

- 输入隐私：除函数输出允许泄露的信息外，其他参与方不能学习某方输入。
- 正确性：诚实方得到正确输出。
- 可用性/公平性：在某些模型下，避免恶意方阻止诚实方获得输出。
- 可验证性：在恶意安全模型下，参与方不能用错误中间值悄悄污染结果。

## 常见协议路线

### 秘密分享

把秘密拆成多个 share，单个 share 不泄露原文，足够数量的 share 才能恢复或参与计算。典型方案包括 Shamir secret sharing 和 additive sharing。加法通常很便宜，乘法需要 Beaver triples 等预处理材料。

一个两方 additive sharing 例子：

```text
x = x0 + x1 mod q
P0 holds x0
P1 holds x1
```

单个 share 看起来随机；只有合并两个 share 才能恢复 `x`。在线计算中，加法可以本地完成；乘法通常要消耗预处理 triple：

```text
a, b, c = a*b
use shares of a,b,c to compute shares of x*y without revealing x,y
```

### Garbled Circuits

由一方把计算电路“加密打包”，另一方在不知道中间明文的情况下求值。常用于两方计算，适合布尔电路和固定函数。

### Oblivious Transfer

OT 是许多 MPC 协议的基础原语。接收方从多个消息中选择一个，发送方不知道选择了哪个，接收方也不知道未选择消息。

### 混合协议

实际系统常把秘密分享、OT、同态加密、零知识证明组合起来。不同子任务选择不同表示，例如用布尔分享处理比较，用算术分享处理线性代数。

## 计算模型与电路表示

MPC 的性能高度依赖函数如何表示：

| 表示 | 擅长 | 不擅长 |
| --- | --- | --- |
| 算术电路 | 加法、乘法、线性代数、统计聚合 | 比较、排序、位操作 |
| 布尔电路 | 比较、分支、bit-level 逻辑 | 大规模矩阵乘法 |
| 固定点编码 | 机器学习、金融数值 | 溢出、精度和截断处理 |
| 查表/混合协议 | 非线性函数、激活函数 | 表大小和泄露控制 |

很多业务函数在普通代码里很自然，但转成 MPC 后代价很高。例如排序、去重、字符串匹配、正则、树模型推理、浮点计算都需要特别设计。

## 离线/在线阶段

高性能 MPC 常拆成：

- **离线阶段**：生成 Beaver triples、OT extension 相关材料、随机掩码等，与真实输入无关。
- **在线阶段**：拿真实输入计算，尽量只做少量通信。

这种拆分的好处是低延迟；风险是预处理材料必须唯一使用，不能复用。复用乘法 triple 可能泄露输入关系。

## 典型协议安全属性

设计或选型时需要明确：

- **Privacy**：参与方从 transcript 中学不到额外输入信息。
- **Correctness**：输出对应正确函数。
- **Robustness**：恶意方中止或发送垃圾时，诚实方能检测。
- **Fairness**：防止一方先拿到输出后中止。
- **Guaranteed output delivery**：即使部分参与方恶意，诚实方仍得到输出。

后两项在不诚实多数场景中通常代价高或不可能完全实现，很多系统只承诺 privacy/correctness，不承诺公平输出。

## 安全模型

MPC 文档中必须先看安全模型：

- 半诚实（semi-honest）：参与方按协议执行，但试图从 transcript 推断额外信息。
- 恶意（malicious）：参与方可任意偏离协议、发送错误消息、选择性中止。
- 诚实多数：只要多数参与方诚实即可保证安全。
- 不诚实多数：即使多数恶意，也能提供隐私/正确性，但成本更高。
- 静态腐化 vs 自适应腐化：攻击者是在协议开始前选择腐化对象，还是运行中动态选择。

很多“高性能 MPC”只在半诚实模型下成立。跨组织生产系统通常需要恶意安全、审计和输入合法性证明。

恶意安全通常需要额外机制：

- MAC/check 信息验证 share 未被篡改。
- ZKP 证明输入满足范围、格式或业务约束。
- Cut-and-choose 检查 garbled circuit 构造。
- 一致性检查防止选择性中止和不一致输出。
- 审计日志记录每轮协议和参与方身份。

## 输入、输出与元数据泄露

MPC 只隐藏协议规定隐藏的内容。仍要单独评估：

- 输出是否足以反推某方输入。
- 小集合 PSI 结果是否暴露成员关系。
- 查询次数是否允许差分攻击。
- 消息大小和轮数是否泄露输入规模。
- 参与方在线时间、失败模式是否泄露业务状态。

因此 MPC 系统常与差分隐私、访问控制、最小阈值、审计和法律协议一起使用。

## 安全边界

MPC 能保护：

- 各方输入不被其他参与方或计算协调方直接看到。
- 中间值不以明文形式集中出现。
- 在合适模型下，恶意参与方难以伪造计算。

MPC 不能自动保护：

- 函数输出本身泄露的信息。
- 参与方提供假输入导致的业务污染，除非有输入证明或外部审计。
- 结果被合法接收方滥用。
- 元数据泄露，例如谁参与、何时参与、消息大小、网络模式。
- 拒绝服务，恶意方可以中止协议。

## 工程实现要点

- 明确有限域/环大小，避免溢出导致错误结果。
- 对定点数截断和舍入做安全协议，不能直接泄露低位。
- 对随机数、OT seed、triple 存储做高强度保护。
- 输入对齐/实体匹配常比核心计算更敏感，PSI 阶段也要建模。
- 网络延迟比 CPU 更常成为瓶颈，跨地域部署要谨慎。
- 开源框架的安全模型、编译器优化和 side-channel 假设要逐项确认。

## 与 TEE 的比较

MPC 不需要信任单个 CPU/云平台，也不会在某个节点出现完整明文输入。但它通常比 TEE 慢、通信量大、开发复杂。TEE 更适合通用 workload 和低延迟服务；MPC 更适合多机构互不信任、法律或商业上不能把数据交给单方的协作。

## 适用场景

- 多银行联合风控或黑名单匹配。
- 医疗机构联合统计。
- 广告转化归因。
- 隐私集合求交（PSI）。
- 跨组织模型评估和特征计算。
